## Why
Goodchat needs a safe, deterministic way for plugins to extend database schema and persist runtime state. Today schema sync is static and plugin hooks have no defined DB capability boundary, which blocks production-ready plugins like rate-limiter and creates risk for future public plugins (name collisions, drift, and unsafe data access). This epic establishes plugin-declared schema, deterministic migration integration, and capability-scoped hook persistence so plugins can ship end-to-end without compromising platform safety.

## What
`01-schema-foundation-drizzle-only`
- [ ] `goodchat db schema sync` generates deterministic consumer-side Drizzle schema files for core/auth/plugins.
- [ ] Workspace packages are not the canonical location for Drizzle table declarations; they are derived artifacts.
- [ ] Core Drizzle dialect-specific schemas are generated by `goodchat cli` from schema DSL files.
- [ ] Auth Drizzle dialect-specific schemas are generated from Better Auth canonical schema import with deterministic normalization.
- [ ] Auth schema artifacts are generated regardless of auth enabled/disabled runtime config to avoid environment-only toggle drift.

`02-contracts-foundation`
- [ ] Plugins can declare schema in `packages/contracts/src/plugins` using a typed logical contract.
- [ ] Contract supports sqlite/postgres/mysql generation without per-dialect hand-authored plugin schemas.

`03-schema-normalization-discovery`
- [ ] `goodchat db schema sync` discovers configured plugins from `src/goodchat.ts` in registration order.
- [ ] Plugin physical table naming is mandatory namespaced by plugin identity, with optional config key suffix for multi-instance disambiguation.
- [ ] Schema sync hard-fails on effective-name collisions or incompatible overlapping definitions; no silent override behavior is allowed.

`04-drizzle-integration`
- [ ] Plugin schema is integrated into the existing Drizzle migration flow (including project patterns used in `examples/minimal/drizzle`), not a separate migration system.
- [ ] Migration flow is explicit and deterministic: schema/migration artifacts are generated and validated before runtime processing starts.
- [ ] `goodchat db schema sync --check` is read-only, fails deterministically on byte-level artifact drift (exit 1), and emits human-readable plus JSON diagnostics.
- [ ] `goodchat db schema sync --check` semantic categories include `ARTIFACT_STALE`, `MISSING_ARTIFACT`, `UNEXPECTED_ARTIFACT`, `PLUGIN_SCHEMA_CONFLICT`, `MIGRATION_HISTORY_DIVERGENCE`, and `NON_DETERMINISTIC_INPUT`.

`05-migration-gating-runtime-readiness`
- [ ] Runtime startup is gated on successful schema validation and migration readiness; if migration fails, hooks/message processing do not start.

`06-hook-db-gateway`
- [ ] Hook contracts in `packages/contracts/src/hooks` expose capability-scoped DB access for plugin hooks, limited to that plugin’s declared tables.
- [ ] Public hook DB APIs do not permit direct cross-plugin/core table access.

`07-rate-limiter-plugin`
- [ ] Rate-limiter plugin declares its schema through the new contract, uses hook DB capability to persist/evaluate limits, and works end-to-end across restart.
